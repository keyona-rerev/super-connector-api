### 1. Events Feature — Complete

**Option C hybrid model confirmed:**
- Interview series episodes remain as **sub-projects** under their initiative
- All other events (panels, workshops, networking) are a new **Events** first-class object

#### Railway — `db.py`
Two new tables added to `init_db()` via `CREATE TABLE IF NOT EXISTS` (auto-creates on next deploy, no migration needed):

**`events` table:**
```
event_id     TEXT PRIMARY KEY  (auto: EVT-{timestamp})
data         JSONB NOT NULL
created_at   TIMESTAMP
updated_at   TIMESTAMP
```
Indexes on `data->>'venture'` and `data->>'event_type'`.

JSONB data fields: `event_id` · `event_name` · `event_type` · `venture` · `date` · `location` · `status` · `initiative_id` · `description` · `notes`

**`event_guests` table:**
```
guest_id    TEXT PRIMARY KEY  (auto: EG-{timestamp})
event_id    TEXT NOT NULL
contact_id  TEXT
data        JSONB NOT NULL
created_at  TIMESTAMP
updated_at  TIMESTAMP
```
Indexes on `event_id` and `contact_id`.

JSONB data fields: `guest_id` · `event_id` · `contact_id` · `full_name` · `role` · `guest_status` · `notes`

New db functions: `upsert_event`, `get_event`, `get_all_events` (filterable by `event_type` and `venture`), `delete_event`, `upsert_event_guest`, `get_guests_for_event`, `delete_event_guest`

#### Railway — `models.py`
Four new Pydantic models: `EventPayload`, `EventStatusUpdate`, `EventGuestPayload`, `EventGuestUpdate`

**Status values:** `Planning` / `Confirmed` / `Complete` / `Cancelled`
**Event type values:** `Hosting` / `Attending` / `Workshop`
**Guest role values:** `Speaker` / `Panelist` / `Attendee` / `Interviewee` / `Organizer` / `Sponsor`
**Guest status values:** `Invited` / `Confirmed` / `Attended` / `No-show` / `Declined`

#### Railway — `main.py` — 10 new endpoints
```
GET    /events                    List all events (?type=Hosting&venture=BTC)
GET    /event/{id}                Fetch event + guests + guest_summary {total, confirmed, attended}
POST   /event                     Create event
PUT    /event/{id}                Update event (merge — omitted fields preserved)
DELETE /event/{id}                Remove event
PATCH  /event/{id}/status         Quick status update
GET    /event/{id}/guests         List guests for event
POST   /event-guest               Add guest to event
PATCH  /event-guest/{id}          Update guest role/status/notes (all optional, PATCH semantics)
DELETE /event-guest/{id}          Remove guest from event
```

`GET /event/{id}` returns event data + full guest list + `guest_summary: { total, confirmed, attended }` in a single call — web app uses this to render the badge without a second fetch.

All 10 endpoints confirmed live in Railway Swagger docs at `/docs`.

## NEXT SESSION: CONTACTS CRM REBUILD

### Problem
The current Search Contacts page is **search-only**. You type a description, it returns vector matches. There is no way to:
- Browse all contacts
- Add a new contact manually
- Edit an existing contact's fields
- Sort or filter contacts into buckets (venture, relationship health, source, etc.)

This is why SXSW connects and other new warm contacts haven't been entered — there's no UI to do it.

### What to Build

**Replace (or restructure) the Search Contacts page into a proper Contacts CRM view.** The semantic search stays — it's valuable — but it becomes one mode within a fuller contacts experience.

#### Layout options to decide at session start:
- **Option A — Two tabs on the same page:** `Browse` tab (paginated grid/list with filters) + `Search` tab (existing semantic search). Simplest lift.
- **Option B — Browse as default, search as topbar action:** Contacts page defaults to browsable grid, search box filters it in real time (client-side on loaded records) or fires a semantic search when you hit enter on a non-trivial query.
- **Option C — Separate nav items:** Keep Search Contacts as-is, add a new `Contacts` nav item that's the CRM view. Most surface area but cleanest separation.

**Keyona's preference should be confirmed at session start.** Lean toward Option A or B — fewest nav items, most coherent.

#### Core capabilities to build:

**1. Browse + Filter**
- Paginated contact grid (use existing `GET /contacts?limit=50&offset=N`)
- Filter bar: venture, relationship_health, activation_potential
- Sort: recently updated (default), name A–Z
- Each card: name, role, org, venture tag, relationship health badge, activation potential badge

**2. Add New Contact**
- `+ New Contact` button in topbar (same pattern as other pages)
- Modal with all ContactPayload fields: full_name, title_role, organization, how_we_met, venture, what_building, what_need, what_offer, relationship_health, activation_potential, notes
- On save: `POST /contact` with auto-generated `contact_id = C{timestamp}`
- Vectorizes automatically server-side

**3. Edit Existing Contact**
- Contact detail drawer (already exists, currently read-only)
- Add Edit button that opens the same modal pre-filled
- On save: `PUT /contact/{id}` — re-vectorizes automatically

**4. Buckets / Sorting**
The "buckets" Keyona wants are filter views, not separate data structures. Useful bucket dimensions:
- **By venture** — ReRev Labs / Prismm / BTC / Sekhmetic / Personal / blank
- **By relationship health** — Strong / Active / Warm / Cold / Unknown
- **By activation potential** — High / Medium / Low / Unknown
- **By source tag** — if a `source` or `tags` field is added to the contact profile (e.g. "SXSW 2026", "BTC Summit")

⚠️ **Source/tags field decision:** The current contact schema has no `source` or `tags` field. At session start, decide whether to add one. If yes, it's a one-line addition to `ContactPayload` in `models.py` and the `ContactPayload` class in `main.py` — no schema migration needed (JSONB). This would let Keyona tag contacts by where she met them (SXSW, event name, referral, etc.) and filter by it.

#### Existing infrastructure to reuse:
- `GET /contacts?limit=50&offset=0` — already built, returns paginated list
- `POST /contact` — already built, vectorizes
- `PUT /contact/{id}` — already built, re-vectorizes
- Contact drawer HTML + JS already exists — extend, don't rebuild
- Contact card CSS already exists in the search results grid — reuse

#### What does NOT need to change:
- Railway API — no new endpoints needed (existing CRUD is sufficient)
- The semantic search functionality — keep it, just make it one mode
